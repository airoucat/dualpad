# DualSense 适配上古卷轴 V 的输入注入重构研究：从 xinput.dll 劫持迁移到原生 InputEvent 桥接

## 背景与现状聚焦

你当前的实现路径是“**HID 采样（hidapi）+ xinput.dll 劫持注入游戏**”，而实际痛点是：一旦把“注入载体”和“输入语义”都绑在 XInput 这条链上，生态里任何一个同样依赖/代理/叠加 XInput 的组件（Steam Input、驱动层虚拟手柄、Overlay、别的 Mod/注入器、甚至同目录的别的代理 DLL）都可能让**按键冲突、重复输入、状态错位**更容易发生（你们已经感受到“冲突加剧”）。  

这类问题的本质不是“你们的 HID 解析不对”，而是“XInput 这条通路在 Skyrim 里是一个**容易被多方同时占用**的共享入口”。因此你提出的迁移方向——把输入语义切到 Skyrim 自己的 **InputEvent / ControlMap** 管线——在工程上是成立的，并且能把冲突面显著缩小到“游戏内部的统一输入分发点”。（后文会用 Skyrim 的类型/结构证据展开。）

同时，你明确强调“只在主线程构造并注入 InputEvent，不在 HID 线程碰游戏对象”，这是非常关键的工程约束：即使是插件作者也会在代码里注明 `UI::GetSingleton()` 等对象“可能不是线程安全的”，提醒不要跨线程触碰运行时对象。citeturn6view5

## Skyrim 的原生输入形态：InputEvent 链、语义字段与设备编码

要论证“**HID → 主线程注入 InputEvent**”是否可行，必须先把 Skyrim 里 InputEvent 的“形状”讲清楚。CommonLibSSE（以及其 NG 分支）对多数输入相关结构已做了逆向封装，能直接作为类型证据。

### InputEvent 是单链表节点而不是“单个事件”

`RE::InputEvent` 的核心特征是：  
它带有 `device`（输入设备类型）、`eventType`（事件类别）以及 `next` 指针，显式构成**单链表**的一段。也就是说，Skyrim 每帧向下游分发的不是“一个事件”，而是一条 `InputEvent*` 链。citeturn17view0  

`InputEvent` 的事件类型枚举里明确包含 `kButton`、`kThumbstick` 等（与你方案里要先落地的 ButtonEvent / ThumbstickEvent 对齐）。citeturn17view0

### ButtonEvent 的“按下/抬起/长按”语义是由 value 与 heldDownSecs 组合表达

`RE::ButtonEvent` 继承自 `RE::IDEvent`，其成员只有两个浮点：`value` 与 `heldDownSecs`，并由一组内联判断函数派生出 IsDown/IsHeld/IsUp 等语义。citeturn20view0  

这点直接支持你方案里提到的“PressedEdge / ReleasedEdge / Held”的状态机层：在 Skyrim 侧，事件对象本身的语义就是这么表达的，不是 XInput 那种“快照式 state”。citeturn20view0

### ThumbstickEvent 明确区分左右摇杆，并携带 x/y 浮点

`RE::ThumbstickEvent` 同样继承自 `IDEvent`，内部用 `InputType` 区分左右摇杆（`kLeftThumbstick = 0x0B`，`kRightThumbstick = 0x0C`），并携带 `xValue` / `yValue`。citeturn21view0  

这意味着你在主线程构造摇杆事件时，完全可以按 Skyrim 期望的结构填充（而不是把摇杆硬塞进 XInputGetState 的轴值语义里）。

### IDEvent 同时具有 userEvent 与 idCode：这是你未来“绕开 ControlMap”的关键抓手

`RE::IDEvent` 在 `InputEvent` 基础上增加了两类核心信息：  

1) `BSFixedString userEvent`：`QUserEvent()` 将返回它；  
2) `std::uint32_t idCode`：`GetIDCode()` 返回它。citeturn22view0  

这非常重要，因为 Skyrim 的输入处理既可能看 `idCode`（更接近“按键码/设备码”），也可能看 `QUserEvent()`（更接近“动作名/语义事件名”）。在实践里，很多对输入做逻辑判断的钩子/插件会直接使用 `QUserEvent()` 与 `UserEvents` 中的动作名做比较，例如 Sprint 逻辑就见过这种写法。citeturn30search4

### 设备类型 INPUT_DEVICE：你注入时需要选对 device

`INPUT_DEVICE` 的枚举（键鼠手柄/虚拟键盘等）在 CommonLibSSE 中明确给出：`kKeyboard`、`kMouse`、`kGamepad` 等。citeturn23view1  

这会直接影响：ControlMap 映射、不同输入上下文的优先级、以及菜单态/战斗态的处理分支。因此你的“桥接层”必须明确选择以 `kGamepad` 注入还是以键盘/鼠标注入。

### Skyrim 的 gamepad idCode 不是 XInput VK，而是 BSWin32GamepadDevice::Key

Skyrim 里 Win32 手柄设备的“按键码”来自 `RE::BSWin32GamepadDevice::Key`。其枚举明确列出方向键、Start/Back、肩键、A/B/X/Y 等。citeturn32view0  

更关键的是：现成插件在处理 `ButtonEvent` 时，会把 `idCode` 强转成 `BSWin32GamepadDevice::Key` 来解释，这基本等价于告诉你“**游戏内的 button idCode 就是这个枚举域**”。citeturn31view0turn32view0  

因此你在构造 `ButtonEvent` 时，把 `idCode` 填成 `BSWin32GamepadDevice::Key::kA / kLeftShoulder / ...` 这一类值，是与 Skyrim 语义对齐的路径。citeturn32view0turn31view0

## Native InputEvent Bridge 的可行实现：主线程并链注入的工程化细节

你提出的 **Native InputEvent Bridge（NIEB）** 分层（采样层/状态机层/主线程桥接/交给原生系统执行）总体方向是正确的；下面把“怎么落地、哪里是硬点”拆得更具体。

### 关键目标不是“自己做一套输入系统”，而是把 DualSense 输入变成 Skyrim 能吃的 InputEvent 链

因为 `InputEvent` 天生就是链表节点（带 `next` 指针），你要做的“桥接”本质上是：在某个稳定的时机，构造一段你自己的 `InputEvent` 子链，然后**并到本帧原有事件链里**（或者生成一个新的头，把原始链挂到你链尾）。citeturn17view0turn22view0  

这与“替换链”相比，风险明显更低：  
你不需要猜测引擎内部还会不会产生日志/系统事件（例如设备连接事件），也不会屏蔽掉玩家真实键鼠输入。

### 并链注入点的现实选择

从“可稳定实现”角度，常见有三条路：

**路径 A：钩住引擎分发前的统一入口，做链表追加**  
你在“本帧输入分发前”拿到 `InputEvent* head`，把你的事件追加上去，再让引擎照常分发。这最符合你原文强调的“并链而不是替换链”。由于 `BSInputDeviceManager` 自身就是输入事件源（`BSTEventSource<InputEvent*>`），它天然处在“要发送链”的位置。citeturn33view0  

**路径 B：注册一个 InputEvent 的 EventSink，尝试在回调里改链**  
一些插件通过 `BSInputDeviceManager::GetSingleton()->AddEventSink(...)` 注册输入事件回调（示例代码里就是这么做的）。citeturn24search1turn31view0turn33view0  
但把“改链”放在 sink 里有两个隐患：  
1) 你不一定能保证 sink 的调用顺序（尤其是你想让 PlayerControls/菜单控制接到“改过的链”）；  
2) sink 获得的是 `InputEvent* const*`（头指针本身不可改），虽然你能通过修改 `next` 来追加，但仍受限于先后顺序与潜在的并发/重入。citeturn31view0  

**路径 C：完全绕开“本帧链”，自己额外发送一条链**  
理论上，你可以构造一条链，然后主动触发一次输入事件分发。但这样更容易出现“同一帧分发两次输入”的不可预期后果（需要你精准确认 PlayerControls 对多次分发是否幂等）。在没有明确证据前，不建议作为首选。

结合你们的目标（彻底摆脱 xinput.dll 注入链、减少 hook 冲突），**路径 A 的“分发前并链点”**仍是最契合 NIEB 的落点。

### 你文中那四个“硬条件”，用结构证据逐条落到实现约束

你给的四条硬条件非常正确；这里把它们与实际结构对齐，明确“踩雷点通常在哪”。

主线程与时机：你必须在“分发前”做并链，否则你只能影响你之后的接收者，而无法保证 PlayerControls/菜单链路一致。`BSInputDeviceManager` 作为 `BSTEventSource<InputEvent*>` 的事实，说明确实存在一个“集中分发输入事件链”的位置。citeturn33view0turn31view0  

内存生命周期：因为链是 `InputEvent*` 节点串起来的（`next`），并且每个节点还有 `userEvent/idCode/value/heldDownSecs/xValue/yValue` 等字段，**任何节点释放或被覆盖都会导致后续遍历踩内存**。结构上 `next` 是裸指针，不存在自动管理。citeturn17view0turn20view0turn21view0turn22view0  

设备/按键语义：你必须把 `device` 设对（例如 `kGamepad`），并把 `idCode` 填入 `BSWin32GamepadDevice::Key` 域，否则下游要么映射不到动作，要么在不同上下文下被当成别的设备事件。citeturn23view1turn32view0turn22view0  

并链而非替换：因为这是单链表遍历（`for (...; one_event; one_event = one_event->next)` 这种写法在插件里非常常见），你替换头/切断会直接丢事件。citeturn31view0turn17view0  

### 推荐的事件构造策略：以“小型事件池”替代临时栈对象

由于 `InputEvent`/`ButtonEvent`/`ThumbstickEvent` 都是简单 POD 风格结构（成员字段少，且链依赖裸指针），为了避免“临时对象生命周期”踩雷，最稳妥的方法是：

在桥接层维护一个**每帧复用的事件池**（例如固定数组 + 游标），每帧开始清游标、写入需要注入的事件节点、串起 `next`，分发结束后不立刻 free，而是等待下一帧复用覆盖。  
这种做法更贴近引擎“每帧生成事件链”的模型，也避免你在不知道引擎是否会尝试析构/缓存事件的情况下发生释放策略冲突。（这条是工程推论：结构证据告诉我们链表是裸指针串联，最怕生命周期出错。citeturn17view0turn20view0turn21view0）

### DualSense 的 HID 线程与主线程桥接：你这次要主动“反向隔离”

你已经提出“只在主线程构造并注入 InputEvent，不在 HID 线程碰游戏对象”，这是正确方向。原因并不仅是“线程安全抽象”，而是 Skyrim 运行时对象（UI、PlayerCharacter、ControlMap 等）在很多插件中都被认为“不保证线程安全”，甚至直接在代码里标注提醒。citeturn6view5  

落地建议是：HID 线程只产出“逻辑输入帧”（例如按钮边沿、轴值、手势结果），主线程在稳定注入点消费该帧并生成 `ButtonEvent/ThumbstickEvent`。

## 为什么这条路线能显著降低冲突：从“共享 API 注入点”切到“游戏内部统一事件链”

你们现在的冲突加剧，通常来自两个叠加：

一是 **注入入口** 使用 xinput.dll 劫持：只要别的组件也在这条链上（代理 DLL、Hook、驱动层虚拟 XInput 设备），就会出现“谁先加载/谁先 hook/谁覆盖 IAT”之类的工程性冲突面。

二是 **输入语义** 走 XInput：XInput 的语义更像状态快照，而 Skyrim 内部对很多动作（连按、按住、菜单态输入）更依赖 `ButtonEvent` 的 `heldDownSecs/value` 组合。citeturn20view0  

当你把“输入注入”转为 `InputEvent` 链的并链注入时，冲突面会更集中且更可控：你不再争夺外部 API 的入口，而是把 DualSense 的输入变成 Skyrim **已经要处理的**那种事件节点（`kButton/kThumbstick`），并在遍历链时被自然消费。citeturn17view0turn20view0turn21view0  

但这里有一个必须提前规划的现实问题：如果玩家同时启用了 Steam Input/系统虚拟手柄，Skyrim 仍可能收到“真实 XInput → 内部 gamepad device”产生的事件链，你再注入一套 `kGamepad` 事件就会出现**双倍输入**。这个不是 NIEB 方案的失败，而是“输入源重复”。因此迁移时要同步考虑：是否需要在游戏侧关闭原生手柄输入或对重复源做屏蔽（例如只在检测到 DualSense HID 存在时禁用/忽略系统 gamepad）。`BSInputDeviceManager` 暴露了 `IsGamepadEnabled()` / `IsGamepadConnected()` 等状态查询入口，至少在“策略判定”上你有抓手。citeturn33view0

## 关于你补充的目标：屏蔽玩家自定义 ControlMap、只按系统默认映射

你提出“屏蔽玩家自己在游戏设置里改键位后的 ControlMap，只用系统默认 ControlMap”，从技术上并非做不到，但需要明确你到底想屏蔽哪一层：

### 你真正想要的是“动作语义稳定”，而不是“文件层面禁止改键”

从结构与现有实践看，Skyrim 输入事件里同时存在：

- `idCode`（设备按键码）；citeturn22view0turn31view0  
- `userEvent`（动作名/语义名，`QUserEvent()` 返回）。citeturn22view0turn31view0turn30search4  

并且至少在一些输入处理/钩子实现里，会直接用 `QUserEvent()` 与 `UserEvents` 的字段进行比较来判断动作（例如 sprint）。citeturn30search4  

这带来一个对你非常有利的推论：  
如果你的 NIEB 注入事件能够**稳定填好 `userEvent`（动作名）**，那么下游逻辑就可能更多地依赖动作名而不是玩家改过的键位映射，从而达到“动作语义稳定”的目标，而不必真的去禁止玩家改键。citeturn22view0turn30search4  

换句话说，与其“屏蔽 ControlMap_Custom”，更推荐你把这个目标重述为：

> “我的 DualSense 映射始终对齐一套固定的 **UserEvent 动作语义**；玩家怎么改键位不影响 DualSense 的动作语义。”

这对用户接受度也更高：玩家可以继续改键盘/手柄键位，但你的 DualSense 手势/特殊键仍然映射到你定义的动作层。

### 如果你坚持“只用默认 ControlMap”，可选的工程路线

这里给三条路线（风险由低到高），你可以按是否愿意动引擎内部行为来选：

**路线一：在你注入的 IDEvent 上直接指定 userEvent，弱化对 ControlMap 的依赖**  
你注入的是 `ButtonEvent/ThumbstickEvent`，它们都继承 `IDEvent`，天生带 `userEvent` 字段。citeturn20view0turn21view0turn22view0  
如果你能为每个 DualSense 输入明确指定动作名（例如使用 `UserEvents` 中的 sprint/activate 等字段作为 canonical 名字），则你可以在很大程度上绕开“玩家改键导致 idCode→userEvent 映射变化”的问题。（这是基于结构与现有用法的推论。citeturn22view0turn30search4）

**路线二：启动时快照一份“默认映射表”，运行时逻辑永远用快照**  
`ControlMap` 内部明确包含按输入上下文与设备划分的 mapping 容器（`InputContext` 里 `deviceMappings[INPUT_DEVICES::kTotal]`），并提供 `GetMappedKey()`/`GetUserEventName()` 这类查询 API。citeturn29view0turn29view1turn23view1  
你可以在“默认映射加载完成后、玩家自定义生效前/或生效后但你仍能读到默认”的时机把映射表复制出来，并在你自己的 bridge 层永远使用这份快照。这样不会破坏玩家文件，只是你选择不尊重它。

**路线三：直接干预 ControlMap 的运行时行为（不推荐默认开启）**  
例如拦截 `ControlMap::GetMappedKey()`，强制返回默认映射；或拦截加载自定义映射的路径（具体加载函数与文件位置取决于运行时版本/实现细节）。  
这条路线技术上可行，但兼容性与用户体验风险最大：因为 `ControlMap` 不只是“按键映射”，它还包含 controls enable flags、上下文优先栈等状态，粗暴干预可能影响菜单、控制禁用、文本输入等行为。citeturn29view1  

结论是：**优先做“动作语义稳定（userEvent）”，把“强制默认 ControlMap”做成可选开关**，否则会显著降低 mod 的可接受度与兼容性。
